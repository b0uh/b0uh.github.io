Marketplaces API, the key to onboard merchants

  Tue 02 May 2017

TL;DR: Having a good API will speed your merchant onboarding and time-to-market. On the contrary a bad API or no API can demotivate your merchant and make you and them lose money.



Why do you need a marketplace? ¶

E-commerce has always been a big component of the internet as we know it. Having a website to sell your product is nowaday a requirement for a merchant and there is now a new trend to do marketplaces. Marketplace is simply a website that connect merchants to clients and behave as a trust tier. The most famous examples are Amazon, Zalando or Alibaba. One clue about marketplaces developments is the explosion of marketplace solutions that exists today, just to name a few : Mirakl, Izberg, Wizplace. Most of the merchant are using marketplace solution or custom solution in order to allow other merchant to sell on their own website and make it grow in term of products, traffic and turnover.

Why an API? ¶

So you manage to sign contracts with some merchants who want to sell on your marketplace, congratulation ! Then, what? They will want to put some products quickly then generate and manage some orders. Most of marketplace solutions allow you to do it with their backoffice, so that’s cool, everything can go quite fast. But it require for the merchant to dedicate some time to handle his products on your website and this potentially quite often. He will need to learn a new tool as he need to manage everything from your backoffice. While in the meantime, all merchants have already a software to manage this on their own, this can be an e-commerce solution (Prestashop, Magento, Shopify, ...) or an ERP (SAP, Sage, ...).

The API can help your merchant by allowing him to manage your marketplace from its favorite tool !

However, this cost money, consider it when your marketplace reach a minimum of merchants.

An API for everything, because automation rules ¶

You now have an API for merchant, but does it handle everything the merchant need? For example on most of marketplace you have a list of categories on which the merchant can sell. This should not provided in a CSV file but as an API. Why? Because if tomorrow you want to add new categories with your API, you don’t have to communicate about it and merchants can easily retrieve it. With an API for everything you allow the merchant to automatise everything ! It means less friction points.

What’s a good/bad API? ¶

A good API is a when your merchant is fully autonomous to plug your marketplace to its IT system. From my experience there is only a few marketplaces with good API. Here are some good and bad practices on marketplace API, those criteria are just a few among a long list, took a look at Vinay Sahni post "Best Practices for Designing a Pragmatic RESTful API" for more informations.

Good practices:

  • Having a nice documentation
  • Autonomous merchant (don’t need to contact you for accessing the API for example)
  • Restful API (lien)
  • Standard format (JSON, XML, SOAP)
  • Clear errors (too often the error after an API call is not clear, this will cause the merchant to contact you and therefore lose time)
  • Status page (you will have downtime, communicate about it)
  • API backwards compatibility
  • Bonus: having an SDK

Bad practices:

  • Complexe API (all the complexity should be handle by you, not the merchant)
  • Bad documentation (example, a simple “Javadoc” style documentation)
  • Fuzzy error (it will make lose time to everyone)
  • Breaking changes (no backward compatibility, merchants will have to update their development, they may not have the money/resources and therefore this is a big risk for business)
